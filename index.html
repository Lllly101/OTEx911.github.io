【译】-Imagemagick远程代码执行漏洞

发表于 2017-07-01 | 分类于 Web | 阅读次数

ImageMagick是一款非常出名的软件，主要用于图像的编辑、转化等。ImageMagick为几乎所有的编程语言（PHP、Python、Ruby等等）提供了函数库支持，因为容易上手的特性，大多数开发者在遇到图片操作或剪切需求都会选择ImageMagick。

因为最新版的ImageMagick没有对文件名（文件名通常是传到内部处理外部协议的函数处）进行恰当过滤，所以这会导致攻击者通过上传图片执行远程命令。因为在图片上传过程的完全代码执行，漏洞显得非常严重，所以研究员戏称为ImageTragick而不是CVE-2016-3714。

漏洞细节

自从初步部分披露这漏洞以来，我们的研究团队一直致力于构建可用的poc便于exp的了解和防护测试。历经数小时的研究和在安全社区帮助下，我们对漏洞理解很充分，所以构建了一个使用ImageMagick的简单PHP上传工具和控制服务器的exp。

该漏洞太容易利用了，攻击者只需要使用ImageMagick图像上传工具。在研究过程中，我们发现了大多数流行web应用和Sass产品都存在该问题，接着我们和他们私底下联系希望他们及时打补丁。不幸的是即使有媒体的关注，也总有人未意识到该问题。

接着聊聊更多细节，该漏洞其实可以分为四个小问题，来自Mail.Ru安全团队的Karim非常详细的解释了这个的由来。简单总结一番，下面几点是我们一定得了解的：

  1. 远程代码执行是在.mvg/.svg文件上传功能上发生的。攻击者通过恶意的文件就能在服务器上执行shell命令。这有个简单的例子：

    添加MVG文件，wget命令执行接着pastbin文件的内容会被存在backdoor.pl文件中。

  2. 远程文件删除。攻击者通过ephemeral:/协议可以移除服务器上的文件。

  3. 远程文件移动。与远程文件删除原理相似，只不过是通过伪协议msl:/移动文件。

  4. 使用label:@协议可以披露文件内容。

综合利用上述这些问题，攻击者的攻击面会很广，同时可以通过工具去控制使用ImageMagick的web应用。注意只有MGV后缀过滤不严格，其他任意文件格式都会被检查然后执行文件中的命令。

我认为随着研究人员对ImageMagick的持续关注，ImageMagick中的更多的漏洞迟早会被披露。

请注意ModSecuirty和其他IDS等使用了最新签名套件的工具无法检测或屏蔽这个漏洞。昨天我们更新了防火墙修复这个漏洞，在Sucuri防火墙后的用户是被保护的。同时我们也回溯之前的攻击，结果仍未有发现。随着攻击者拥有了自己的exp，这个局面会发生改变。

保护

尽管使用我们防火墙设备的用户已经能够防御这种漏洞，我们仍然推荐大家遵从ImageMagick开发人员的意见，修改/etc/ImageMagick/plicy.xml文件，禁用图像文件中的MVG，HTTPS，EPHEMERAL和MSL命令。在配置文件的厦门这部分新增：

发表于 2017-06-26 | 阅读次数

本文由高级安全分析师Zach Julian编写。你可以在推特上联系他。

Atmail是一家以云端和本地电子邮件服务著名的供应商。Atmail的客户有普通公司、主机供应商以及包括DreamHost、美国的LegalShield、波斯尼亚的m:tel、iiNet和澳大利亚的Optus互联网服务供应商。

作为DreamHost的atmail用户，在漏洞奖励机制工作期间，我曾经历过数个令人记忆犹新的email跨站脚本攻击，所以我打算在atmail的web客户端挖掘前端漏洞。数小时后，我发现了一个能起作用的payload，但想通过构造一只xss蠕虫进一步利用。xss蠕虫最出名的案例是05年MySpace遭受攻击)，在那之后2014年TwitterDeck遭05年蠕虫变种的攻击。

本文中，我将详细讲述构造通过利用受害者联系人来自我繁殖的XSS攻击。

构建XSS Payload

首先需要创建一个能够在atmail过滤器保持完整的xss payload。以发送一封包含所有有效的html标签的邮件为开始，接着查看哪些标签会在发送后保留下来，最终我决定使用img标签。img标签非常适合构建xss payload，它的优势在于攻击者必须在atmail中点击显示图像才能出发xss。构建更好的payload应是使用一种不需要用户交互、自身能渲染的标签。

接着，我开始关注atmail是如何过滤我的paylod。为了避开过滤器和在受害者浏览器中渲染出语义正确的标签这两个目的，我需要观察atmail是如何操控img标签中的字符和html属性。通过发送包含所有有效属性的img标签（<img src="x" align="left" alt="test" border="1px" crossorigin="anonymous" height="100px" hspace="100px" ismap longdesc="test" sizes="(min-width: 600px) 200px,），我发现只有src、alt、longdesc、style、height和width属性被允许。此外我还发现了payload上的其他的一些变化，比如单引号变成了双引号，onerror事件被移除，没有src属性的img标签被移除。

尽管onerror事件被移除，我却怀疑单引号转为双引号可能会在白名单的绕过上起作用，如果在img标签中同时使用onerror和单引号的话。最终，怀疑被证实是对的，虽然我不得不在两个img标签中使用了一对双引号。有效的xss payload如下：
