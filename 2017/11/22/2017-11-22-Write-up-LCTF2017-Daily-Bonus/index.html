When I was trying to get the flag, the first problem is how to confirm that the param site is parsed by php function parse_url. In fact, on the 2017 Black Hat, Geeker Orange described the symptom when abusing parsers in the back-end server. You can get it from this video.

In a word, developers implemented parse_url , readfile and cURL and so on with their own standard instead of the RFC(3986).

So, I used the following payload to fuzz then see the response from web application.

All these payloads are variations from Orange Paper (Awesome payloads). Till now, I was still far away from the flag. The hint in challenge descriptions is local, then tried to change the schema from http to file.

Everything is still confused except the hint local, the parsers parse_url. If you are careful enough, you’ll find the http://test@127.0.0.1:80@www.baidu.com/test.php/ on the 404 Not Found Page, it adds the slash / to http://test@127.0.0.1:80@www.baidu.com/test.php.

Now let’s add a question mark to the url then access that.

`file://test@127.0.0.1:80@www.baidu.com/ect/passwd?

Then with the response , I changed the path to etc/flag?.

LCTF{1eTus_q14ndao_B4_387t439hg9342}

Summary

The purpose of adding question mark to the end.

You can get answer from RFC(3986) or Orange PPT in 2017 Black Hat. Here is the quote from RFC(3896) section 3.2.

The authority component is preceded by a double slash (“ //“ ) and is terminated by the next slash (“ /“ ), question mark (“ ? “ ), or number sign (“ #” ) character, or by the end of the URI.

You must be careful with everything in the challenge. Don’t let anything suspicious go.

Happy hacking.
